AGOGE ACADEMY API - GUIA DE MODULOS, BRANCHES, COMMITS, MERGE Y VERSIONADO

Objetivo
========

Mantener el proyecto ordenado mientras esta en setup y preparar una forma
repetible de crear modulos, trabajar branches, hacer commits claros, mergear
sin perder historial y versionar main/develop de forma consistente.


1. Ramas principales
====================

main
----
- Rama estable.
- Debe contener codigo probado y listo para release/tag.
- Cada version publicada debe existir como tag: v0.1.0, v0.1.1, etc.

develop
-------
- Rama de integracion.
- Aqui se unen features y fixes antes de pasar a main.
- Durante setup, develop puede avanzar rapido, pero debe mantenerse verde.

Branches temporales
-------------------
- Se crean desde develop, no desde main, salvo hotfix urgente.
- Se eliminan cuando ya estan mergeadas.


2. Nombres de branches
======================

Usa nombres cortos y descriptivos:

feature/auth-login
feature/users-module
fix/ci-prod-start
fix/docker-entrypoint
chore/prisma-config
docs/workflow-guide
release/0.2.0
hotfix/prod-start

Regla practica:

tipo/scope-descripcion-corta

Tipos recomendados:
- feature: nueva funcionalidad.
- fix: bug o error.
- chore: mantenimiento interno.
- docs: documentacion.
- release: preparacion de version.
- hotfix: correccion urgente desde main.


3. Flujo para crear una branch nueva
====================================

Antes de empezar:

git checkout develop
git pull origin develop
git status

Crear branch:

git checkout -b feature/nombre-del-modulo

Ejemplos:

git checkout -b feature/auth-module
git checkout -b fix/ci-prod-start
git checkout -b docs/workflow-guide


4. Estructura recomendada para un modulo nuevo
==============================================

Los modulos de dominio deben vivir en:

src/modules/<nombre-del-modulo>/

Estructura base:

src/modules/users/
  users.module.ts
  users.controller.ts
  users.service.ts
  dto/
    create-user.dto.ts
    update-user.dto.ts
    user-response.dto.ts
  entities/
    user.entity.ts
  interfaces/
    users-repository.interface.ts
  repositories/
    users.prisma-repository.ts
  tests/
    users.service.spec.ts

Para modulos mas simples, minimo:

src/modules/<modulo>/
  <modulo>.module.ts
  <modulo>.controller.ts
  <modulo>.service.ts
  dto/
  tests/

Reglas de arquitectura
----------------------

- Controllers: reciben HTTP, validan DTOs y delegan.
- Services: contienen reglas de negocio.
- DTOs: entrada/salida de datos y validaciones.
- Repositories: acceso a base de datos o APIs externas.
- Entities/interfaces: tipos propios del dominio.
- Common: solo helpers compartidos por varios modulos.
- Config: solo configuracion de entorno o bootstrap.
- Database: solo Prisma y conexion a base de datos.

Importante:

- No mezclar logica de negocio en controllers.
- No usar Prisma directo en controllers.
- No crear carpetas compartidas antes de que realmente se compartan.
- Cada modulo debe exportar solo lo necesario.


5. Crear un modulo Nest
=======================

Opcion CLI:

yarn nest generate module modules/users
yarn nest generate controller modules/users
yarn nest generate service modules/users

Luego organiza DTOs, repositories y tests dentro de la carpeta del modulo.

Antes de commitear:

yarn format
yarn lint
yarn build
yarn test:ci
yarn prisma:validate

Si el modulo toca endpoints HTTP, tambien:

yarn test:e2e


6. Commits claros
=================

Este proyecto usa Conventional Commits y commitlint.

Formato:

tipo(scope): descripcion en minuscula

Ejemplos validos:

feat(auth): add login endpoint
fix(ci): correct production start entrypoint
fix(docker): align runtime entrypoint
docs(release): document version 0.1.2
chore(database): add prisma seed script
test(users): add users service coverage

Tipos permitidos:

feat, fix, docs, style, refactor, perf, test, chore, revert, ci, build

Scopes permitidos actualmente:

auth, users, storage, analytics, common, shared, config, database, deps,
docker, github, release, ci, docs

Si necesitas un modulo nuevo con otro scope, agrega ese scope primero en
commitlint.config.js dentro de scope-enum.

Cuando hacer commits
--------------------

Haz commits pequenos:

1. Codigo del modulo.
2. Tests del modulo.
3. Documentacion si aplica.
4. Version/changelog si aplica.

Evita commits mezclados como:

"cambios varios"
"fix"
"updates"


7. Versionado
=============

El proyecto sigue Semantic Versioning:

MAJOR.MINOR.PATCH

Ejemplos:

0.1.2
0.2.0
1.0.0

Mientras el proyecto esta en setup:

- PATCH: fixes de setup, CI, docs, dependencias, Docker.
  Ejemplo: 0.1.1 -> 0.1.2

- MINOR: nuevo modulo base o funcionalidad inicial.
  Ejemplo: 0.1.2 -> 0.2.0

- MAJOR: cambios incompatibles cuando ya exista una version estable.
  Ejemplo: 1.0.0 -> 2.0.0

Archivos que se actualizan al versionar:

package.json
CHANGELOG.md

Despues del merge a main:

git tag -a v0.1.2 -m "Release v0.1.2"
git push origin v0.1.2


8. Merge correcto entre branches
================================

Flujo normal de feature a develop:

git checkout develop
git pull origin develop
git checkout feature/nombre
git rebase develop

Resolver conflictos si aparecen.

Luego:

git checkout develop
git merge --ff-only feature/nombre
git push origin develop

Si --ff-only falla, significa que la branch no esta lineal con develop.
En ese caso vuelve a:

git checkout feature/nombre
git rebase develop


9. Pasar develop a main
=======================

Cuando develop esta verde:

git checkout main
git pull origin main
git merge --ff-only develop
git push origin main

Crear tag de version:

git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z

Sincronizar develop si main recibio algun hotfix:

git checkout develop
git merge --ff-only main
git push origin develop


10. Hotfix urgente desde main
=============================

Usar solo cuando main esta roto o hay una correccion urgente:

git checkout main
git pull origin main
git checkout -b hotfix/descripcion-corta

Corregir, probar y commitear:

yarn lint
yarn build
yarn test:ci

Luego:

git checkout main
git merge --ff-only hotfix/descripcion-corta
git push origin main

Sincronizar develop:

git checkout develop
git merge --ff-only main
git push origin develop

Crear tag patch:

git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z


11. Limpiar branches usadas
===========================

Cuando la branch ya esta mergeada:

git branch -d feature/nombre
git push origin --delete feature/nombre

Ver branches remotas:

git ls-remote --heads origin

Ver estado local:

git branch --all --verbose --no-abbrev


12. Checklist antes de push
===========================

Antes de hacer push:

git status
yarn format:check
yarn lint
yarn build
yarn test:ci
yarn prisma:validate

Si toca HTTP o bootstrap:

yarn test:e2e

Si toca dependencias:

yarn audit --level moderate


13. Checklist para revisar Actions
==================================

Si falla CI:

1. Leer el job exacto que falla.
2. Reproducir el comando localmente.
3. Revisar si el comando depende de dist, Docker, DB o secrets.
4. Corregir el script local primero.
5. Ejecutar build/test local.
6. Subir commit pequeno con scope ci, docker, deps o config.

Ejemplo:

Error:
Cannot find module dist/main

Causa:
El build genera dist/src/main.js.

Fix:
Usar node dist/src/main.js en package.json, Dockerfile y docker-compose.yml.


14. Regla de oro
================

Una branch debe responder una sola pregunta:

"Que problema o feature resuelve esta branch?"

Si la respuesta tiene "y tambien", probablemente deben ser dos branches o dos
commits separados.
